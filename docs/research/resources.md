## TJ Pitre / Southleft — AI-ready design systems & audit tooling

- **Machine-Readable, Not Machine-Governed: What Config 2026 Actually Proved** — https://southleft.substack.com/p/machine-readable-not-machine-governed · TJ Pitre · 2026-07-06 · Agents should read/query the system; humans keep the judgment layer (deprecation, membership, meaning). Cites Wolosin's JSON-vs-Markdown ~80% token-efficiency result and Astryx's dense/MCP surface. · *Positioning language for ds-bench (validation tooling for the judgment layer); receipt for structured-metadata checks.*
- **Context-Based Design Systems: A New Model for the AI-Driven Product Lifecycle** — https://southleft.com/insights/design-systems/context-based-design-systems-a-new-model-for-the-ai-driven-product-lifecycle/ · TJ Pitre · 2025-07-04 · CBDS encodes "what it means, how it behaves, when it should be used"; each lifecycle step inherits context; upstream flaws (bad naming, missing states, lack of intent) cascade. Proposes the "context engineer" role. · *Umbrella framing; "lack of intent" maps to usage-guidance category.*
- **Anatomy of an AI-Ready Design System** — https://southleft.substack.com/p/anatomy-of-an-ai-ready-design-system · TJ Pitre · 2026-05-07 · `[paywalled]` Thesis (free preview): "The practices that make a design system AI-ready are the same ones that make it readable, period." Promised hierarchy of practices not recovered. · *Independent statement of ADR 0003's intrinsics-dominate thesis.*
- **Designed vs. Declared: How to Document a Design System for AI** — https://southleft.substack.com/p/designed-vs-declared-how-to-document · TJ Pitre · 2026-05-22 · Declared meaning (description fields, annotations, metadata) beats designed meaning (visual doc frames) for agents: "Inferred meaning is where agents get expensive and unreliable." · *Justifies weighting metadata-carried guidance over doc-frame heuristics in docs/guidance checks.*
- **My Beef with Agentic Design Systems** — https://southleft.substack.com/p/my-beef-with-agentic-design-systems · TJ Pitre · 2026-06-19 · Critique of "agentic" terminology: autonomous loops without human gates produce "confident, unowned drift"; test = "what rejects the agent's output, and who decided the rule?" · *Terminology guidance: stick with "agent-readiness", avoid "agentic design system".*
- **"AI just turns the lights on" — TJ Pitre on Design Systems and AI** — https://www.designsystemscollective.com/ai-just-turns-the-lights-on-tj-pitre-on-design-systems-and-ai-d57b095e9a90 · Shane P Williams (interviewer) · 2026-03-28 · Five machine-readability layers (structured tokens, complete metadata, articulated intent, component contracts, validation tooling); "machine-readability is structural integrity"; failure list (hard-coded values, incomplete variants, naming inconsistency, missing behavioral metadata, drift, vague docs). · *Closest thing to a published dimension list from Southleft; maps ~1:1 to ds-bench categories.*
- **AI-Ready Design Systems w/ TJ Pitre (Product Craft Podcast)** — https://arseniharkunou.substack.com/p/ai-ready-design-systems-w-tj-pitre · Arseni Harkunou (host) · 2026-04-08 · Readiness = structure and cleanliness, not tools; "check engine light" metaphor; "prompting isn't the hard part," context quality is. · *Supports intrinsics-over-delivery weighting.*

## Diana Wolosin / Indeed — machine-readable metadata & MCP benchmarking

- **How Indeed Made Their Design System Machine-Readable for MCP and LLMs** — https://intodesignsystems.substack.com/p/ai-design-system-mcp-example · Sil Bormüller (guest post, Into Design Systems) · 2026-06-08 · `[paywalled]` `[secondary]` Wolosin's benchmark: 22 prompts × 3 runs × 2 × 8 MCP configs = 1,056 prompts over 77 components; formats Markdown / plain MD / MD+JSON / JSON / TOON; JSON = 80% fewer tokens, 5× lower cost ($300 vs $1,500/yr) vs Markdown (~30k tokens/query, 82% coverage, hallucinations). Numbers recovered via the author's free article below. · *The quantitative receipt for structured-metadata checks; format choice is a performance characteristic.*
- **Your Design System Is Not Ready for AI Agents** — https://www.intodesignsystems.com/blog/design-system-not-ready-for-ai-agents · Sil Bormüller · 2026-04-10 · Five failure modes: documentation drift, Markdown-without-benchmarking, no trust levels, MCP without always-on rules, monolithic component definitions. Progressive disclosure = always-on rules + on-demand MCP + AGENTS.md. "Plant seeds, not trees." · *Best free source for the Indeed numbers; candidate check: foundations coverage in agent context files.*
- **Machine-Readable Design Systems: Designing for AI as a User** — https://www.designsystemscollective.com/machine-readable-design-systems-designing-for-ai-as-a-user-28077c9f2144 · Diana Wolosin · 2026-03-05 · Defines machine-readable DS ("express its knowledge in structured formats AI can interpret directly"); coins AIX; notes retrieval alone still yields typography/spacing/color drift. · *Definition source; the drift observation motivates always-on-foundations checking.*
- **AI Metadata: Powering a Design System MCP** — https://www.designsystemscollective.com/ai-metadata-powering-a-design-system-mcp-b5deafcae8f5 · Diana Wolosin · 2025-08-11 · Three metadata layers (behavioral rules, business intelligence, implementation props); JSON as "the core language of AI"; GitLab-webhook freshness sync. · *Schema precedent for what manifest coverage should contain; freshness-sync supports staleness checks.*
