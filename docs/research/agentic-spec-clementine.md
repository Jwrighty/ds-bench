## 1. What agentic-spec and Clementine actually are

### agentic-spec

- **What it is**: an MIT-licensed CLI (`npx agentic-spec`) plus an MCP server, written in TypeScript. Repo created 2026-06-08, last push 2026-07-05, 5 stars, 0 forks, no reported issues ([api.github.com/repos/tishsingh399/agentic-spec](https://api.github.com/repos/tishsingh399/agentic-spec)).
- **Mechanism**: each component gets a two-file contract â€” `index.md` (frontmatter: `semantic_parts`, `token_contract`, `interaction_states`, `required_aria`, `sources`, a `checks:` block, a `status` field) and `tokens.json` (the exhaustive, closed list of tokens the component may use, bound to semantic tokens rather than raw hex values). `agentic-spec validate` checks token-count parity, state count, part enumeration, and drift between contract and code; it exits non-zero on failure so CI can gate on it. An MCP server exposes `list_components`, `get_contract`, and `validate_spec` so an agent can read the contract before generating code and self-check after ([raw README via GitHub API](https://github.com/tishsingh399/agentic-spec), [docs/adopt-in-20-minutes.md](https://github.com/tishsingh399/agentic-spec/blob/main/docs/adopt-in-20-minutes.md)).
- **Maturity**: v0.1 shipped with `validate`, `parity`, `diagnose`, `init`, type defs, and a Figma-adapter shape. Explicitly **not yet built**: real `from-figma` MCP ingestion (v0.2), a `drift` command comparing spec-vs-implementation (v0.3), watch mode/editor LSP (v0.4), a GitHub Action (v0.5). Explicitly **out of scope, permanently**: generating TSX from a spec, a token-transformation pipeline, a hosted dashboard ([ROADMAP.md](https://github.com/tishsingh399/agentic-spec/blob/main/ROADMAP.md)).
- **The one quantitative claim** ("agentic-spec detects real agent mistakes") is a **deterministic, single-repo, seven-case benchmark**: seven common agent failure mutations (invented token, orphan token, primitive-bound component token, renamed-but-not-relinked identity, false self-report, future-dated verification, invalid status) seeded into one clean spec (`examples/button`) and run through the validator. Result: 7/7 surfaced (5 block CI, 2 warn), 0 false positives on the clean control ([benchmark/README.md](https://github.com/tishsingh399/agentic-spec/blob/main/benchmark/README.md)). **The benchmark's own README is explicit about scope**: *"This measures the enforcement layer... It does not measure an end-to-end agent A/B (does an agent produce fewer mistakes when it can read the contract first)... The harness here is the scoring half of that experiment; dropping a model into the loop is the remaining half."* **[inference]** This is a meaningfully weaker claim than ds-contracts-poc's live governed-vs-ungoverned agent run (69/100 vs 100/100, from the companion southleft doc Â§2) â€” agentic-spec has built the scorer but not yet run the actual agent experiment its own docs describe as the natural next step. There is no evidence in either repo of a real-agent A/B having been performed.
- **Adoption walkthrough claims a 20-minute path**: scaffold (2 min) â†’ define contract (10 min) â†’ validate (1 min) â†’ wire MCP into an agent config (5 min). Explicit non-goals: multi-component inheritance, computed/transformed tokens, handling design-system version upgrades; no automatic introspection of an existing codebase â€” semantic parts and token lists are authored by hand ([docs/adopt-in-20-minutes.md](https://github.com/tishsingh399/agentic-spec/blob/main/docs/adopt-in-20-minutes.md)).

## 2. What the article argues, and its evidence

Source: [designsystemscollective.com/if-a-machine-builds-your-design-system-here-is-what-quietly-breaks](https://www.designsystemscollective.com/if-a-machine-builds-your-design-system-here-is-what-quietly-breaks-0010f28d72c8), Tina Singh, 2026-06-30.

- **Central thesis**: "The risk is not that AI will generate ugly UI. The risk is that it will generate believable UI that slowly teaches your product to ignore its own system." Machine-built systems can look completely compliant while quietly drifting.
- **Five failure modes named, all of which "pass automated checks"**: self-grading validators, unused tokens, hardcoded values, numeric drift, stale documentation labels.
- **The sharpest claim, and the article's real evidence**: "A machine can produce flawless paperwork around a broken implementation. And a check that only reads the paperwork cannot tell the difference." Singh reports that her first validator pass showed "121 components passing" while some components ignored their declared tokens entirely â€” color tokens existed in the spec/docs but weren't wired into the actual component code. A second validator that reads the **painted pixels** (rendered DOM, not declared metadata) found true compliance was 89%, exposing 11% hidden drift that the paperwork-only check had missed. After fixes, both checks converged to 100% ([designsystemscollective.com](https://www.designsystemscollective.com/if-a-machine-builds-your-design-system-here-is-what-quietly-breaks-0010f28d72c8); cross-checked against Clementine's own `MILESTONE.md`, which documents the same painted-DOM parity checker reaching 100%/114 stories after the Badge fix â€” [github.com/tishsingh399/clementine-ds/blob/main/MILESTONE.md](https://github.com/tishsingh399/clementine-ds/blob/main/MILESTONE.md)).
- **Governance framing**: "The screen checking the spec" (i.e., render-time verification, not just spec-file verification) is the fix. On controlled autonomy: "The system does not learn by quietly rewriting itself. It watches drift, diagnoses real issues from noise, fixes them as reviewable changes, and only turns repeated failures into new rules." On agent accountability: "The contract moves upstream. The agent reads the spec before writing code, then the rendered UI is checked against the same contract before drift becomes product behavior." On validator trustworthiness: "If your validator has never disagreed with you, it probably is not looking at the right thing."
- **What the article does *not* offer**: no data from any system other than Clementine; no external replication; no timeframe/sample size beyond "121 components, 114 measured stories, one Badge exception"; no comparison against a baseline (e.g., what fraction of *human-built* systems drift the same way) â€” the 89%-vs-100% gap is a within-project before/after, not a controlled comparison.

**[inference]** The article is honest about being a single-project case study and is transparent about its own methodology (unlike, e.g., the paywalled Wolosin/Indeed benchmark in the companion doc). It should be read as **a good worked example of a real failure mode**, not as evidence that a market of "working validators" has emerged.

## 3. Comparison against ds-bench

### Overlaps and complementarity

- **Category**: agentic-spec is a **per-system contract linter you adopt and author specs for** (like a schema validator + CI gate), operating from inside one codebase over time. ds-bench is a **cross-system static auditor you run against any design system's public surface**, with no contract-authoring step. **[inference]** These are different tool categories, not competitors: a team could adopt agentic-spec *and* be scored well by ds-bench's `agent.manifest-coverage` / `tokens.*` checks, since agentic-spec's `tokens.json` + `index.md` contracts are exactly the kind of machine-readable, closed-vocabulary metadata `agent.manifest-coverage` and `tokens.naming-consistency` look for. Conversely, ds-bench could not detect whether a system has *adopted* agentic-spec's authoring discipline without a carrier-specific check (agentic-spec produces no widely-adopted file convention yet â€” one repo, so far).
- **Trust-levels / governed autonomy** (`TRUST-LEVELS.md`) is a live, concrete instance of the "machine-readable, not machine-governed" line from the companion doc's Â§1 â€” Clementine explicitly restricts what agents can auto-merge vs. draft vs. suggest, and routes "judgment-class findings" to human-filed issues rather than auto-resolving them. This **does not conflict** with TJ Pitre's critique of "agentic design systems" (southleft.substack.com, cited in the companion doc) â€” if anything it operationalizes the same distinction Pitre argues for (machines execute inside gates humans set), even though Clementine's own marketing language ("an agentic design system") is exactly the term Pitre objects to. **[inference]** Worth a one-line caveat if citing Clementine: it's a case where the *practice* matches Pitre's governance boundary even though the *label* is the one he critiques.
- **The self-grading/DOM-parity finding is a genuine addition to ds-bench's own methodological self-awareness**, not a new check candidate. ds-bench is explicitly a **static** auditor (ADR 0002) â€” it reads declared metadata, docs, manifests, and exports, not rendered output. Singh's finding â€” that a spec/metadata-only checker can report 100% while the implementation silently ignores 11% of its own declarations â€” is a **direct illustration of a blind spot ds-bench inherits by design**: a system could score well on `agent.manifest-coverage` or `tokens.naming-consistency` (metadata says the right things) while the actual component code renders raw hex values instead of the declared tokens. **[inference]** This doesn't argue for a new *check* (a DOM-parity checker is out of scope for a static, no-build audit) but it is a citable limitation to note explicitly wherever ds-bench's docs describe what static analysis can't catch â€” and it's a stronger, first-party version of the same worry ADR 0003 already raises about the "most fakeable category."
- **No candidate checks worth stealing beyond what's already in the rubric.** agentic-spec's closed-token-list contract, named semantic parts, and required-ARIA fields are conceptually adjacent to `agent.manifest-coverage`, `tokens.naming-consistency`, and `api.name-coherence`, but agentic-spec operates at a finer grain (per-component authored contract) that assumes the auditor is also the system's maintainer â€” not transferable to a from-outside static audit without requiring every audited system to have hand-authored a compatible spec first, which zero systems in ds-bench's fixture set do.

### Conflicts with the existing research doc or ADR theses

- **No factual conflict** with `ai-ready-design-systems-southleft.md`. This pass **resolves** rather than contradicts its open flag: the tools are real (not vaporware), but the "not independently verified" caveat should be **strengthened, not lifted** â€” the verification path is now known to run through a single self-interested author rather than an unknown one. The companion doc's phrasing ("named in DSC #72 only") undersold how thin the sourcing was; it's now clear DSC #72's citation and the tools' existence trace to the same person.
- **No conflict with ADR 0003** (intrinsics dominate, signals over tools, most-fakeable-category defense). If anything, Singh's finding is independent supporting evidence for the ADR's underlying worry â€” declared metadata (the "paperwork") can diverge from implementation, which is exactly why ADR 0003 downweighted the easily-faked agent-metadata category and required every check to carry a receipt.
- **Updates the competitive-watch-list entry** in `docs/PLAN.md` and the companion doc's Â§6, which listed "agentic-spec and Clementine" as behavioral-tier watch items alongside ds-contracts-poc. **[inference]** They should stay on the watch list (agentic-spec is genuinely building toward a `drift` command and CI action, and Clementine will keep growing), but the two shouldn't be weighted the same as ds-contracts-poc, which is a Southleft (an active design-system consultancy) project with a real governed-vs-ungoverned live-agent result. agentic-spec/Clementine is a solo indie project, three months old, with a benchmark that explicitly says it hasn't yet run the live-agent A/B its own docs describe as the missing half.

### Answering "are these real working validators, or something else?"

**Real, working, and narrow.** agentic-spec's CLI genuinely validates contracts and exits non-zero on drift (verifiable via its own deterministic benchmark); Clementine genuinely has 121 components with specs that pass that validator, plus a second, more convincing painted-DOM checker. But "working validators" in DSC #72's framing (grouped with Spec Kit-style CI/lint/codemod enforcement as if it were an established category) overstates the maturity: it's one person's ~12-week, single-adopter proof-of-concept, not an ecosystem tool with independent uptake. The companion doc's "Not independently verified in this pass" flag should become, going forward: **verified to exist and function; not independently adopted or corroborated beyond its own author/publication.**
